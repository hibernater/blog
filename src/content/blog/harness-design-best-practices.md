## 一、为什么需要 Harness

Vibe Coding 长任务有一个根本矛盾：

> **AI 是无状态的，但代码库是有状态的。**

这个矛盾不解决，长任务必然出问题。常见的失败模式：

- **Context Amnesia**：早期说过"不能用 SQLite"，第40轮 Agent 建议用 SQLite
- **Architectural Drift**：第3小时的代码和第1小时的架构开始脱节
- **Happy Path Bias**：只做"现在跑通"的路径，边界条件和可维护性被忽略
- **Error Cascade**：早期小错误没被发现，后续代码层层叠在上面
- **Overfit Fix**：修 bug 时 special-case 硬编码，而非修根因

Harness 是在 AI 无状态的现实下，**强制注入工程纪律**的机制。

---

## 二、常见误解：三元组不够

很多项目的 Harness 是这样的：

```
CLAUDE.md + 阶段 + 测试
```

这个三元组有用，但只解决了部分问题。它保障的是：规则有没有、步骤对不对、代码跑没跑通——但管不到两个更深的问题：

1. **约束靠自律执行**：规则写了，但没有工具强制检查，等于建议
2. **验证者和执行者是同一个 Agent**：自己验证自己，系统性偏弱

## 四、完整 Harness：5个元素，5个不同的问题

讨论下来，够用且不繁琐的 Harness 是这5个元素，每个解决一个不同的问题，职责不重叠：

```
CLAUDE.md   → 知识持久化（规则 + 决策 + 状态，跨会话不丢）
阶段        → 任务分解与节奏控制（防止 AI 一口吃个胖子）
门禁        → 约束机械化执行（不靠自律，靠工具卡）
测试        → 行为正确性验证（代码做没做到规格要求）
独立 Judge  → 架构/决策独立验证（隔离上下文，防自我确认偏差）
```

### CLAUDE.md 三层结构

CLAUDE.md 包含三类更新节奏完全不同的内容，这是分层的理由：

```
规则层（强约束 + 软约束）  ← 很少变
决策层（为什么这样做）     ← 每次重要判断后追加
状态层（现在是什么样）     ← 每个阶段结束后刷新
```

规则层内部区分强/软约束（无需再分层，标注即可）：

```markdown
## 规则

### 强约束（门禁执行）
- 不允许硬编码密钥          ← pre-commit 检查
- 测试通过率 > 80%          ← CI 阻断
- Controller 不能直接调 DB  ← 架构测试检查

### 软约束（Agent 理解执行）
- 页面风格温暖自然，非电商风
- 空状态必须有温度感文案
```

强约束 = 可以被工具机械化检查，必须在门禁中有对应执行。
软约束 = 无法自动化，只能靠 Agent 理解。

**规则层和门禁的关系**：规则层是规则库（知识），门禁是执行机制（流程控制）。规则层告诉 Agent 约束是什么，门禁确保约束没被违反才能继续走。

## 五、CLAUDE.md 的三层结构

CLAUDE.md 在 Harness 里的职责只有一件事：**跨会话的知识持久化**，让下一个 Agent 启动时不是从零开始。

基于这个职责，它需要包含三类性质不同的内容，三类内容的**更新节奏完全不同**，这是分层的理由：

### 规则层
**是什么**：这个项目"不能做什么、必须怎么做"

**更新节奏**：很少变，一旦确定轻易不改

**内部分两类**（无需再分层，标注区分即可）：
- **强约束**：可以被工具机械化检查，对应门禁有执行
  - 例：不允许硬编码密钥 / 测试通过率 > 80% / Controller 不能直接调 DB
- **软约束**：需要 Agent 理解执行，无法自动化
  - 例：页面风格温暖自然，非电商风 / 空状态必须有温度感文案

### 决策层
**是什么**：记录"为什么这样做"，不是"做了什么"

**更新节奏**：每次做重要判断时追加，只增不改

**格式**：`[日期] 做了什么 | 为什么 | 考虑过的替代方案`

**触发时机**：技术选型、架构边界调整、重要 trade-off

**存在的理由**：架构决策只活在对话上下文里，对话结束即消失。这是跨会话漂移的根因。有了决策层，下一个 Agent 知道"为什么选了 X 而不是 Y"，不会在新会话里推翻之前的合理决策。

### 状态层
**是什么**：现在代码库是什么样的

**更新节奏**：每个阶段结束后刷新

**内容**：实际目录结构 + 核心模块功能摘要 + 已知问题

**关键原则**：尽量从代码/测试输出自动生成，而非 AI 手写。AI 手写的状态可信度打折扣——容易省略负面信息，或与实际代码不一致。

---

## 六、门禁

**是什么**：约束的机械化执行，执行下一步的前置检查

**和规则层的关系**：规则层定义约束是什么（知识），门禁负责检查约束有没有被违反（执行）。规则层是规则库，门禁是执行机制，不是同一个东西。

**实现形式**：
- pre-commit hook：提交前自动跑测试和约束检查，不过不让提交
- CI pipeline：push 前必须绿灯
- 阶段跳转必须显式确认，不允许 Agent 自己说"通过了"就过

**核心价值**：把"靠 AI 自律遵守规则"变成"工具强制不让违反规则"。没有机械化执行的约束，本质上是建议。

---

## 七、独立 Judge

这是7个项目里 7/7 缺失的元素，也是当前 Harness 设计最大的结构性缺陷。

**为什么 Executor 不能兼任 Judge**：执行过程产生的上下文会污染判断。写了这段代码的 Agent，天然倾向于认为这段代码是对的。这不是某个 Agent 的问题，而是系统性偏差。

**Judge 的核心要求：隔离上下文**。Judge 不能看 Executor 的对话历史，只看代码、测试结果、规格文档。

**三种实现形式**（按成本升序）：

1. **CI 作为机械 Judge**：测试通过率、lint、不变量检查——客观、零成本，但只能验证预定义规则
2. **LLM-as-Judge**：独立的 LLM，从干净上下文启动，只读代码和规格，不看实现过程——适合阶段验收
3. **人工 Judge**：只看测试输出和 diff，不看 AI 的解释，给出明确的 pass/fail——适合涉及安全/数据库 schema 的关键节点

实践建议：三种组合使用。每次提交跑 CI Judge，每个阶段结束跑 LLM-as-Judge，关键检查点过人工 Judge。

### LLM-as-Judge 的落地设计

**触发时机**：按阶段触发。每个阶段是一个完整的功能模块，颗粒度合适，Judge 有足够上下文判断"这个模块整体对不对"。Agent 自检清单勾完、说"完成"之后自动触发，Judge 通过才能解锁下一阶段。

**Judge 的上下文（只能看这三样，不能多）**：
- CLAUDE.md 规则层
- 这个阶段改动的文件（git diff）
- 测试运行结果（不是测试代码，是运行输出）

**不能看**：Executor 的对话历史、Agent 的自我解释。一旦看了，Judge 就被"实现过程的合理化叙事"污染。

**Prompt 结构**：

```
你是一个独立的代码审查者，你不知道这段代码是如何实现的。
只根据以下规格，判断代码是否符合要求。

[规格：CLAUDE.md 规则层]
[代码变更：git diff]
[测试结果：测试输出]

判断以下问题：
1. 是否有违反强约束的地方？
2. 这个阶段的验收清单，有没有"自检通过但实际没过"的项？
3. 有没有明显的技术债需要记录？

给出 pass/fail + 具体条目。fail 必须说清楚是哪条没过。
不给建议，只给判断。
```

**Judge 的核心价值**：识别"自检通过"和"真正完成"之间的灰色地带。

以余田里阶段五为例：Agent 自检显示"5个阶段全部通过"，但实际上有3件事待人工确认（AppID配置、阿里云部署、微信开发者工具验证）。CLAUDE.md 规则层写着"不能声明完成除非当前阶段自检清单全部通过"——Judge 读到这里，应该对"最终交付"给出 fail，因为依赖真实环境的验证项不能由 Agent 自行勾选通过。这正是 Executor 自检发现不了、而独立 Judge 能发现的典型问题。

## 八、一句话总结每个元素的必要性

| 元素 | 解决什么问题 | 没有它会怎样 |
|------|------------|------------|
| CLAUDE.md | 跨会话知识持久化 | 每次新会话从零开始，规则/决策全部丢失 |
| 阶段 | 任务分解与节奏 | Agent 一口气做太多，出错难以定位和回滚 |
| 门禁 | 约束机械化执行 | 规则靠自律，等于没有规则 |
| 测试 | 行为正确性验证 | 不知道代码有没有做到规格要求 |
| 独立 Judge | 决策/架构独立验证 | 自己验证自己，系统性偏弱，架构腐化无人发现 |

---

## 九、核心设计原则

> **好的 Harness 不是让 AI 更聪明，而是让 AI 的错误更难犯、更容易被发现、更容易被回滚。**

1. **约束必须机械化**：强约束写在规则层，门禁来执行。没有对应门禁的强约束等于软约束
2. **Judge 必须独立**：Executor 兼任 Judge 是系统性问题，不是个人问题
3. **决策必须持久化**：只存在于对话上下文里的决策，下次会话就不存在了
4. **状态尽量自动生成**：AI 手写的状态，可信度低于从测试输出自动生成的状态

## 十、与 Anthropic 文章的对照（v1.1 增补）

2026 年 3 月，Anthropic Labs 团队的 Prithvi Rajasekaran 发表《Harness design for long-running application development》，记录了他用 GAN 思路（generator + evaluator）做长任务编码的工程实践。整篇读完，**和咱们 v1.0 强共识 4 条，差异 5 条**。共识部分（独立评估者、任务分解、结构化交接、客观可验证标准）说明双方走到了同一个结论——这本身是重要的交叉验证。这里展开 4 个对咱们方法论有补强意义的差异。

### 10.1 Harness 是动态的，随模型能力浮动

Prithvi 最有冲击力的观点：

> 每个 harness 组件都编码了一个"模型做不到 X"的假设。模型升级了，这些假设值得重新压力测试，因为它们容易快速过时。

他从 Opus 4.5 → 4.6 时，直接砍掉了 **context reset** 和 **sprint 分解** 两个曾经"必备"的组件。Opus 4.6 的长上下文 retrieval 和持续 agentic 能力提升后，这两个组件从"必需"变成"过度工程"。

**咱们的盲点**：v1.0 把"五元组"当作永久必备，没强调随模型升级要主动审视、做减法。

**新增纪律**：
- 每次大模型升级（不只是 minor）后，主动审视 harness
- 试着拿掉一个组件跑一次实验，看输出质量有没有降
- 没降 → 这个组件已经过时，删掉减负
- 降了 → 保留，但记录"为什么这个模型还需要它"
- **一次只拿掉一个组件**，否则分不清谁在 load-bearing

### 10.2 Evaluator 本身需要 tuning

Prithvi 反复 tune evaluator 才让它停止"包容"。原话：

> Out of the box, Claude is a poor QA agent. In early runs, I watched it identify legitimate issues, then talk itself into deciding they weren't a big deal and approve the work anyway.

**LLM-to-LLM 包容偏差是真实存在的**——Judge 也是 LLM，倾向于对其他 LLM 的产出宽容。

**咱们 v1.0 的盲点**：给了 Judge prompt 模板就完事，没提"Judge 本身要校准、可能仍有偏差需要持续监控"。

**新增 Evaluator 校准流程**：
1. 跑几轮让 Judge 自由打分
2. 读 Judge 日志，标出"它给过但我会给不过"的案例
3. 加 few-shot 例子到 Judge prompt，明确"这种情况要 fail"
4. 提高 skepticism 阈值：在 prompt 里加"宁可错杀，不可放过；存疑时默认 fail"
5. 持续监控，定期抽查 Judge 的 pass 决策

校准是常规动作，不是一次性的。

### 10.3 Sprint Contract — 开工前协商"done 的标准"

Prithvi 的三 agent 系统里，每个 sprint 开始前 generator 和 evaluator 文件化协商完成标准，**双方同意后才开始写代码**。

**为什么有效**：用户的 spec 通常是模糊的（"做一个游戏编辑器"）。
- 直接让 generator 干 → 它做自己理解的版本
- 直接让 evaluator 评 → 它按自己理解的标准评
- 两边理解错位 → 来回返工

**对比咱们 v1.0**：咱们的"验收清单"是单向的（人或 planner 提前写好），没有这种双向协商。Sprint Contract 升级为双向，解决了"spec 模糊 → 双方理解错位"的问题。

**实操流程**：
```
1. Generator 提案：本 sprint 我要做什么、怎么验证完成
   → 写入 sprint-N-contract.md
2. Evaluator 审：方案合理？测试点够？覆盖了哪些 spec？
   → 写入 sprint-N-feedback.md
3. 来回直到双方同意（通常 1-2 轮）
4. Lock contract → Generator 开始写代码
5. Evaluator 按 contract 验收
```

### 10.4 主观任务的评分化方法

代码任务有 verifiable check（测试通过），但**设计 / 文档 / 战略类任务**怎么 Judge？

Anthropic 的 4 维度评分法：

| 维度 | 含义 | 举例（前端设计） |
|---|---|---|
| **Quality** | 整体是否融贯 | 配色/排版/留白是否形成统一气质 |
| **Originality** | 是否有原创性，避免 AI 套路 | 不要紫色渐变+白色卡片这种 telltale sign |
| **Craft** | 技术执行力 | 字体层级、间距、对比度 |
| **Functionality** | 可用性 | 用户能否理解和操作 |

**两个关键技巧**：
- **不平均加权**：Prithvi 把 Quality 和 Originality 权重提高，因为 Claude 在 Craft / Functionality 上默认就 OK，瓶颈在前两项
- **措辞会塑造输出**：在 criteria 里写 "the best designs are museum quality"，generator 真会产出博物馆级设计——criteria 不只是评分标准，也是导向

**咱们 v1.0 的覆盖**：主要针对代码工程任务，没覆盖主观任务的 Judge 方法。v1.1 补上这一块，适用场景：
- 设计类（UI/UX、PPT、海报）
- 文档类（技术文档、战略文档、营销文案）
- 策略类（投资建议、产品规划、商业方案）

## 十一、潜在分歧：门禁 vs Evaluator Agent

读完 Anthropic 文章，一个细微差异是**重心不同**：

| | 咱们 v1.0 | Anthropic 文章 |
|---|---|---|
| **机械错误（密钥/风格/架构边界）** | pre-commit / CI 自动阻断 | 没特别强调 |
| **判断错误（架构合理性、UX 自然度）** | 独立 Judge | Evaluator agent |

**判断**：两者互补不冲突。门禁防机械错误（密钥泄漏、风格违规、依赖关系），成本低、确定性强；Evaluator agent 防判断错误（架构合理性、UX 自然度、主观品味），成本高但能覆盖门禁覆盖不了的"软维度"。

**完整 Harness 应该两者都有**——这一条作为 v1.0 五元组的细化保留。

## 十二、v1.1 核心更新摘要

| 议题 | v1.0 立场 | v1.1 增补 |
|---|---|---|
| Harness 组件是否永久必备 | 五元组永久必备 | 随模型能力浮动，每次升级主动审视 |
| Judge prompt 写完是否就行 | 给了 prompt 模板就完事 | 需要 few-shot 校准 + 持续监控 LLM-to-LLM 包容偏差 |
| 验收标准如何确立 | 单向（提前写好清单） | 双向（Sprint Contract，开工前协商） |
| 主观任务怎么 Judge | 未覆盖 | 4 维度评分 + 非平均加权 + 措辞导向 |

> **核心信念不变**：Harness 让 AI 的错误更难犯、更易发现、更易回滚。
> **新加纪律**：Harness 自身也是动态的，要持续审视、校准、迭代。
