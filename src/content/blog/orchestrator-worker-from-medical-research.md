> 起点是给妈妈做糖尿病循证研究，终点是想清楚了一个被很多人误用的 Agent 模式。

## 故事

今天上午，我让山鸡（我的 Hermes Agent）给妈妈做一份糖尿病晚间飙糖的循证研究 —— 这是个有清晰约束的问题：62 岁女性，HbA1c 7.78%，CGM 28 天 TIR 仅 57.9%，合并颈动脉斑块 + 心舒张减退 + HPV58 阳性，三联口服药仍控不住晚间血糖。

山鸡的做法是把这件事**委派**给一个独立的子 Worker 跑 —— 他自己留在主对话里继续负责"读档案 / 定研究方向 / 写贴冰箱版"。Worker 在隔离上下文里跑了 177 秒，产出 10510 tokens、覆盖 5 个核心问题、每条结论带 PMID 的循证材料。

这个分工模式在 Agent 圈叫 **Orchestrator + Worker**。听起来很简单：主管派活儿、小工干活。但实战下来我发现，**大多数人对它的理解是错的**。

---

## 错觉：O+W 像微服务架构

第一直觉是把它类比成微服务 RPC：主调用方 → 服务发现 → Worker 执行 → 返回结果。

但 Agent 世界有一条铁律：

> **微服务以"功能"区分，Agent 以"上下文"区分。**

我用的这个 Worker，**它不是"循证研究"这个功能**。它是"在妈妈这套约束下做循证检索"这个**临时上下文**的承载者。下次研究爸爸的高血压，会另起一个 Worker —— 虽然它的"技能名"也叫循证研究，但**上下文不同 = 它是另一个 Agent**。

这条差别决定了所有后续设计。

---

## 三个反直觉的设计决策

### 1. Worker 不该有"自主性"

`delegate_task` 默认创建的是 `leaf` Worker —— 它**不能再 spawn 子 Worker**。

很多人觉得"让 Worker 自己开 5 个线程并行查 5 个方向多酷啊"。错。

5 个方向之间**没有依赖**，并行不并行不影响产出质量，但 Worker 自己开子线程**立刻引入协调成本**：要发明 Judge 机制（Cursor 已经踩过坑）、要解决子结果汇总的一致性、要处理某个子线程失败的回滚。

**ROI 为负的复杂度，就是错的复杂度。**

什么时候才让 Worker 当 orchestrator？任务里的子问题**互为输入**时：评估一家公司值不值得投，财务数据要驱动产品问题、产品问题要驱动管理层访谈 —— 这时候 Worker 需要自己穿针引线，才该升 orchestrator。

### 2. 真正的杠杆是"裁剪上下文"

我给 Worker 的 context 只有几百字 —— 用户是谁、不许编造、关键临床数据、输出格式。

**我没让 Worker 自己去读妈妈完整的健康档案。**

为什么？因为档案 80% 内容跟"晚间飙糖"无关。让 Worker 读全档，等于让他**重新过滤一遍我已经过滤过的信息**，白烧一遍 token。

这是 O+W 模式最被低估的环节：**Orchestrator 用领域知识把任务降维，Worker 在干净的小上下文里全速跑。**

Cursor 的 Planner 角色印证了这一点：Planner 的核心价值不是"分发任务"（这是顺带的副产品），而是**"裁剪上下文"**。它知道哪些是噪音、哪些是信号 —— Worker 不需要这个知识，所以 Worker 跑得快。

很多人做 multi-agent，是把任务原封不动甩给 Worker。Worker 等于在重做一遍主管的工作。**这种 O+W 没有杠杆，纯粹是把同样的活儿换个进程跑而已。**

### 3. 产出是"指针"，不是"对话"

Worker 把 10k tokens 的循证材料**直接落盘**到妈妈目录，回传给主线程的只有"文件路径 + 5 个一句话摘要"。

主线程接下来要做的是写"贴冰箱版"，**根本不需要原文 75 条循证**，只需要知道：研究完了、底稿在哪、有哪些反常识结论值得反映到日常方案里。

这呼应了 Anthropic Harness 三层架构（Session / Harness / Sandbox）里 **Session = append-only log** 的精神：**中间产物落盘，主线程只持有指针。**

如果 Worker 把 10k tokens 全文回传，主线程的 context window 会被一次性吃掉一大块 —— 后面继续做事的预算就没了。

---

## 这次还暴露了一个 Hermes 的真问题

第一次跑 `delegate_task` 在 52 秒被中断了。`delegate_task` 是**同步阻塞**调用，被父 turn 的资源上限管着 —— 父 turn 一旦超时，子 Worker 就被砍。

这暴露了 Hermes 当前 O+W 缺一层：**持久化 Worker**。

现在只有两个极端：
- `delegate_task`：同步、易中断、产出立即可用
- `cron`：异步、定时触发、不适合 ad-hoc 场景

**缺中间态**：「一次性、异步、长跑、完成即推送」。

Anthropic Code SDK 的 `subagent` 概念，大概就是补这个空。这也是为什么 Claude 团队近期在 Managed Agents 上投入很多 —— 真正的生产级 Agent 需要这层。

---

## 最大的收获

**裁剪 ≥ 调度。**

如果你在做多 Agent 系统，先问自己一个问题：**Orchestrator 在做调度，还是在做裁剪？**

调度容易（路由表 + 任务队列就够了），裁剪难（要领域知识、要判断力、要克制把所有上下文塞下去的本能）。

容易的部分一抓一大把开源方案，难的部分才是产品的护城河。
