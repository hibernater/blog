# 山鸡周报：从任务助手到主动认知补全系统

> **周期**：2026-05-09 ~ 2026-05-15  
> **关键词**：第二大脑、主动认知补全、健康管家、投资体系、微信读取、双 Mac 基建、Agent 工程、Mac GUI 自动化

这一周表面上做了很多事：整理目录、跑 cron、读微信、分析血糖、查投资信号、写博客、调 Hermes、配 Peekaboo、做第二大脑。但底层其实是一件事：

> **把山鸡从“能完成任务的 AI 助手”，推进成一个能长期维护齐晓宁个人操作系统的私人 agent 底座。**

而今天又多了一层关键升级：

> 山鸡不只是根据你问什么来回答什么，而是要根据你的目标、责任和当前认知结构，主动判断你“需要知道但可能还不知道什么”。

这意味着第二大脑不再只是信息库，也不只是 AI 日报，而开始变成一个 **主动认知补全系统**。

---

## 1. 第二大脑：从知识库升级为主动认知雷达

这一周最重要的变化，是第二大脑开始成型。

我们搭起来的不只是几个文件夹，而是一套持续演化机制：

- `second-brain`：第二大脑主体
- `cognitive-reference-v1.md`：认知参考框架
- 每日雷达：持续扫描外部世界
- 周度循环：整合、反思、抽象
- 跨层综合：打通 AI、投资、工作、健康、agent 工程
- 月度 meta 迭代：定期反省整个认知系统本身

一开始，每日雷达还偏像“AI 前沿摘要”。今天我们把它升级了：

> **每日雷达不是 AI 新闻摘要，而是跨领域的认知补全雷达。**

它应该覆盖五类信息：

1. AI 产品 / 工程 / Agent / Harness / business-os
2. 投资 / 产业链 / 宏观
3. 健康 / 家庭健康管理
4. 组织管理 / 战略 / 商业模式
5. 个人认知系统 / 第二大脑 / 学习方法

并且每条信息不能只是“发生了什么”，而要回答：

- 为什么你需要知道
- 它验证了什么
- 它修正或推翻了什么
- 你需要做什么
- 是否应该进入 inbox / wiki / 投资分析 / 健康档案

今天最关键的共识是：

> **第二大脑不能只根据你和山鸡最近聊什么来调权重。**

因为你不会主动聊自己不知道的东西。如果系统只根据交互频率来判断重要性，就会越来越陷入回音室：

> 你不知道的东西，你不会提；  
> 你不提，系统就以为不重要；  
> 系统不推，你就更不知道。

所以第二大脑后续要采用三层权重：

1. **目标权重**：你现在承担什么责任，正在做什么决策
2. **认知缺口权重**：你没聊，但按目标和角色应该知道什么
3. **交互反馈权重**：你明确说有用/没用，用来校准，但不能封闭视野

一句话：

> **山鸡不是追着你聊过的东西跑，而是要主动补你没看见但应该看见的世界。**

这一周博客体系也真正启动。

已经沉淀或整理的方向包括：

- Harness 设计最佳实践
- Agent 架构：Orchestration vs Choreography

- 从医疗研究任务抽象 Orchestrator / Worker 架构

这些不是普通日志，而是把我们踩过的坑、形成的判断，转成可复用、可分享的思想产品。

比如第二大脑这条线，本来只是解决“山鸡如何长期变聪明”的内部问题。但写成博客后，它变成了一个更通用的问题：

> **AI agent 的长期价值，不在于单次回答有多聪明，而在于它能否形成可持续演化的认知结构。**

Harness 也是一样。我们逐渐形成的判断是：

> **好的 Harness 不是让 AI 更聪明，而是让 AI 的错误更难犯、更容易被发现、更容易被回滚。**

这些内容后续都可以成为你讲 1688、讲 AI 组织转型、讲 agent 工程方法论的素材库。

## 3. 双 Mac / iCloud：把山鸡的底座统一起来

这一周把 `~/Documents/hermes/` 统一成两台 Mac 的共享入口。

这件事看起来像整理文件，但对长期 agent 很关键。之前的问题是：

- Mac Mini 和笔记本各有一套目录
- 有些文件在这台，有些在那台
- 有些是真目录，有些是软链
- cron 在 Mac Mini 跑，工程又在笔记本做，路径容易乱

现在收敛成一个规则：

> **Hermes 产生的共享文件，都写入 `~/Documents/hermes/`，这个目录是两台 Mac 共同指向 iCloud 的单一真相源。**

目前收拢进去的内容包括：

- `wiki/`：agent-engineering、work、reading、ai-news、investment、second-brain、mind-wiki
- `blog/`：博客 markdown 和 html
- `家庭健康/`：晓宁、妈妈、爸爸三人的健康档案
- `投资/`：投资机会池等材料
- `hermes历程/`：山鸡成长与系统演进记录

还建立了“双 Mac 一致性巡检”cron，每周自动检查关键共享目录是否一致。

这一步的本质，是给山鸡补上了文件系统层面的长期稳定性。

## 4. Agent 工程与 Harness：从项目经验到方法论

这一周继续梳理 Cursor 里的项目：

- `business-os-0408`
- `business-os-0408-harness`
- `bussiness-os-cam`
- `MyAgents-research`
- `harness-test-4.2`
- `1688 / industrial-ai`

最后确认了一件事：

> **这些不是零散 demo，而是在围绕一个大问题展开：如何为中小企业构建真正可运行的 AI agent 产品？以及为了让 AI 长任务可靠完成，需要怎样的 Harness 工程框架？**

这一周还结合 Anthropic 的 harness 思路，补充了几个重要观点：

1. Harness 不是固定模板，要随模型能力动态调整
2. Evaluator / Judge 也需要校准
3. Generator 和 Evaluator 开工前需要 Sprint Contract
4. 设计、文档、战略这类主观任务，也要尝试评分化和校准
5. 真实运行验证不能缺，必要时 evaluator 要真正点击应用、观察结果

原来的 Harness 五元组继续进化：

- 持久化上下文
- 阶段化任务
- 自动门禁
- 测试验证
- 独立 Judge
- 动态调整
- Evaluator 校准
- 真实运行验证

## 5. 健康管家：从档案管理到持续跟踪

健康线这一周进入了持续跟踪状态。

妈妈的 CGM 血糖数据开始形成轻量工作流：

> **截图 → OCR → CSV 入库 → 趋势分析 → 周报 → 复诊问题清单**

这比抓包、接口逆向、多机部署都更轻，也更适合家庭健康管理场景。

已经整理了 5/4–5/13 的血糖趋势，关键发现包括：

- 5/11、5/12 连续达标，是值得复盘的好样本
- 5/5、5/10 是明显异常日
- 5/13 早餐后飙糖，可能与黎明现象、早餐结构或用药节奏有关
- 后续复诊应重点询问格列齐特、二甲双胍服用时间、GLP-1 RA / SGLT2i 等方案

家庭健康档案也已经迁入：

```text
~/Documents/hermes/家庭健康/
```

这意味着健康管家不再只是“提醒你注意身体”，而是开始维护真实档案、真实数据和连续决策记录。

## 6. 投资体系：从持仓分析到机会发现流

投资线这一周继续扩张。

原来的投资体系偏“持仓管理”：分析 GDS、HSAI、VOR、TIGR 等持仓，做周报和盘后监控。

这一周进一步补上了“机会发现流”：

- `supply-chain-tracker` 扩展到 AI 基建、半导体、网络安全、AI 应用、云计算等赛道
- 投资周报加入产业链信号
- 补充 X / Twitter 和雪球头部信号源
- 建立传统价值委员会和颠覆式成长委员会两套分析框架

中间也踩了一个重要的坑：一开始有些头部账号没有经过充分验证。你明确指出：

> **投资相关信息不能编造，不确定就必须说不确定。**

这是一个关键约束，因为投资 agent 最危险的不是“不知道”，而是“不知道却装知道”。

## 7. 微信和 Mac GUI：开始接入真实工作流

这一周还开始把山鸡接入真实工作流。

微信侧，用 `wechat-cli` 打通了本地微信读取，成功读取微信群「以交易为生，获财富自由」的聊天记录，并提炼出群内关于：

- 阿里
- 中概
- AI 算力
- 光模块
- 腾讯 AI 一号位传闻
- 阿里裁员赔偿

等讨论。

同时也确认：

- `wechat-cli` 只能读，不能发
- 主动发微信不能靠它
- 后续要走 Mac GUI 自动化
- 发送类动作必须加二次确认，避免误发

Mac GUI 侧，Peekaboo 权限也基本跑通。虽然 macOS 的权限列表对 Homebrew CLI binary 很不友好，`peekaboo permissions` 自检也不完全可信，但最终实测已经可以：

- 截屏
- 移动鼠标
- 点击
- 键盘按键

这意味着山鸡不只是能在终端里读写文件，也开始具备触达桌面应用的能力。

## 8. 山鸡自身：从任务集合到长期系统

最后，山鸡自身这一周也在升级。

我们梳理了所有 cron 定时任务，发现健康、投资、AI、第二大脑、系统巡检已经形成长期节奏。

模型侧，也确认当前主模型为：

```text
gpt-5.5 / openai-codex
```

并配置了 Claude fallback：

```text
claude-opus-4-6 → claude-sonnet-4-6
```

这意味着在主模型触达上限、连接错误或多次重试失败时，山鸡可以自动降级，而不是直接中断。

同时，山鸡成长档案和全景总览也继续更新，记录这一周从“能跑任务”向“能维护系统”演进。
